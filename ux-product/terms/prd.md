# PRD（Product Requirements Document，產品需求文件） — 把「要做什麼」寫清楚的說明書

> **白話說：** 就像你請裝潢師傅來裝修房子之前，寫一份詳細的需求單：「客廳要木地板、牆壁要淺灰色、廚房要裝抽油煙機、插座要裝在這個位置。」不寫清楚，師傅做出來的一定跟你想的不一樣。

---

## 它到底是什麼？

PRD 是 PM（產品經理）寫給設計師和工程師的「功能說明書」。

一份 PRD 通常會包含：
- **背景**：為什麼要做這個功能？（通常連結到 [User Research](user-research.md) 的發現）
- **目標**：做完之後要達到什麼效果？
- **[User Story](user-story.md)**：使用者是誰、想做什麼、為什麼
- **功能細節**：每個畫面要長什麼樣、有哪些欄位、按鈕按下去做什麼
- **邊界條件**：什麼情況下不處理、什麼輸入是不合法的
- **成功指標**：怎麼判斷這個功能做得好不好（例如轉換率提升 10%）

PRD 不是「靈感」或「隨口說說」——它是一份要讓所有人看了就知道要做什麼的文件。

## 生活比喻 / 實際例子

想像你要在蝦皮開一家新店，請朋友幫你架設：

**沒有 PRD 的情況：**
「幫我弄一個好看的賣場，賣衣服的，要有購物車。」
→ 朋友做完，你說「這不是我要的」→ 改了三次還不對 → 兩人吵架

**有 PRD 的情況：**
- 首頁要有大圖 Banner，放當季主打款
- 商品分類：上衣、褲裝、配件，各自一頁
- 每件商品要有 3 張圖、價格、尺寸表、庫存數量
- 結帳流程：選商品 → 選尺寸 → 加入購物車 → 填地址 → 付款
- 付款方式：信用卡、超商取貨付款
→ 朋友照著做，一次就到位

實際場景：
- PM：「這個功能我已經寫好 **PRD** 了，大家看一下有沒有問題。」
- 工程師：「**PRD** 裡沒寫如果使用者沒填必填欄位要顯示什麼訊息。」
- 設計師：「根據 **PRD** 的需求，我畫了三個版本的設計稿，大家投票選一個。」

## 為什麼要知道這個詞？

- PRD 是產品開發流程中最重要的文件之一——寫得好，團隊做事有依據；寫不好，整個專案會走偏
- 跟 [Product Roadmap（產品路線圖）](product-roadmap.md) 的關係：Roadmap 是大計畫，PRD 是每個功能的詳細規格
- 跟 [User Story（使用者故事）](user-story.md) 的關係：User Story 是 PRD 裡面的需求單位
- 如果有人說「先寫 PRD 再開工」，代表他們希望先把需求定義清楚，不要做到一半才發現方向錯了
