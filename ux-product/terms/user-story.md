# User Story（使用者故事） — 用一句話描述「誰想做什麼事、為了什麼目的」

> **白話說：** 就像點餐時跟店員說：「我是上班族（身分），想要一個快速又吃得飽的午餐（需求），因為我午休只有一小時（原因）。」一句話讓對方知道你要什麼、為什麼要。

---

## 它到底是什麼？

User Story 是一種用「使用者的角度」來描述需求的格式。

它有一個固定的句型：

**「身為＿＿＿（角色），我想要＿＿＿（功能），這樣我就能＿＿＿（目的）。」**

例如：
- 身為一個**網購買家**，我想要**看到商品評價**，這樣我就能**決定要不要買**。
- 身為一個**外送員**，我想要**看到客戶的門牌號碼**，這樣我就能**快速找到送餐地點**。
- 身為一個**新註冊會員**，我想要**用 Google 帳號直接登入**，這樣我就**不用再記一組新密碼**。

User Story 的重點不在「怎麼做」（技術方案），而在「為什麼做」（使用者需求）。

## 生活比喻 / 實際例子

想像你要幫家人規劃出遊：

- 媽媽的 User Story：「身為一個怕熱的人，我想要去有冷氣的室內景點，這樣我就不會中暑。」
- 小孩的 User Story：「身為一個愛玩水的小朋友，我想要去有泳池的飯店，這樣我就能玩整天。」
- 爸爸的 User Story：「身為一個開車的人，我想要目的地有停車場，這樣我就不用繞來繞去找車位。」

有了這些 User Story，你就能規劃出「大家都滿意的旅行」，而不是只考慮一個人的需求。

實際場景：
- PM 寫給工程師：「這個功能的 **User Story** 是：身為一個忘記密碼的使用者，我想要用手機收驗證碼重設密碼，這樣我就能快速登入。」
- 設計師：「回到 **User Story** 來看，使用者的目的是快速結帳，所以步驟越少越好。」
- Scrum 會議：「這張 **User Story** 的驗收條件是什麼？」

## 為什麼要知道這個詞？

- User Story 是敏捷開發（Agile）裡最常用的需求描述方式
- 它幫助團隊「從使用者角度出發」，而不是從技術角度出發
- 跟 [Pain Point（痛點）](pain-point.md) 的關係：好的 User Story 通常來自真實的痛點
- 跟 [PRD（產品需求文件）](prd.md) 的關係：PRD 是完整規格書，User Story 是其中的需求單位
