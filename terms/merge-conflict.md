# Merge Conflict（合併衝突）— 兩個人改到同一個地方，電腦不知道聽誰的

> **白話說：** 你和同事各自改了企劃書的同一段話，合併時老闆問：「到底要用誰的版本？」——這就是 Merge Conflict。

---

## 它到底是什麼？

當兩個人同時改了**同一個檔案的同一個位置**，[Git](git.md) 在 [merge](merge.md)（合併）的時候就會發現：「等等，這兩個版本不一樣，我不知道該用哪一個。」

這時候 Git 就會停下來，標記出有衝突的地方，讓工程師手動決定：
- 用 A 的版本？
- 用 B 的版本？
- 還是兩邊都保留，自己整合一個新版本？

Merge conflict 不是 [bug](bug-debug.md)，也不是壞掉了——它只是 Git 在跟你說：「這邊我無法自動處理，請你來決定。」

## 生活比喻 / 實際例子

**合寫一份企劃書：**
- 你和小王各自改了企劃書的「市場分析」段落
- 你寫了「目標客群是 25-34 歲」
- 小王寫了「目標客群是 18-30 歲」
- 要合併成一份時 → 衝突了！到底用誰的？
- 得找人來看一下，決定最終版本 → 這就是**解 conflict**

**你什麼時候會聽到：**
- 「我 merge 的時候有 **conflict**，要花點時間解。」
- 「你先 [pull](pull.md) 最新的再開始改，不然等一下 **conflict** 會很多。」
- 「這個 **conflict** 不複雜，我五分鐘就能解完。」

## 為什麼要知道這個詞？

- 當工程師說「在解 conflict」，就是在處理兩個人改到同一處的問題，需要一點時間
- Conflict 不代表有人做錯事——團隊協作本來就會發生
- 越常 [pull](pull.md) 最新版本、越常把自己的修改 merge 回去，conflict 就越少、越小
- 如果工程師因為「解 conflict」而延遲了一點，是正常的
