# Staging / Production — 彩排和正式演出

> **白話說：** Staging 是彩排場（內部測試用），Production 是正式舞台（客戶看到的）。

---

## 它到底是什麼？

用舞台劇來比喻：

- **Staging（測試環境）**= 彩排——演員可以犯錯、忘詞、重來，觀眾看不到
- **Production（正式環境）**= 正式公演——觀眾在看，不能出包

```
排練室（Development）    彩排場（Staging）      正式舞台（Production）
工程師自己開發測試    →  內部團隊驗收確認    →  所有使用者看到的
可以隨便改            →  接近正式版的環境    →  出問題就是大事
```

### 為什麼要分這麼多環境？

| 如果沒有 Staging | 有 Staging 的好處 |
|-----------------|------------------|
| 新功能直接上正式版，壞了客戶馬上受影響 | 先在彩排場跑一遍，確認沒問題再上 |
| PM 沒機會先驗收 | PM 可以在 Staging 先看過、給回饋 |
| 出 bug 只能在正式環境修，壓力山大 | 大部分問題在 Staging 就抓到了 |

## 日常場景

- **「先上 staging 測一下，沒問題再推 production」**
  → 先彩排一次，確認沒問題再正式公演
- **「Production 出事了！」**
  → 正式演出的時候舞台塌了——最緊急的狀況
- **「staging 環境跟 production 不一樣」**
  → 彩排場地跟正式舞台的設備有差異，導致彩排 OK 但正式演出出問題

> 💡 記住這個鐵律：**永遠不要直接在 Production 上面改東西。** 就像不會在正式演出時臨時改劇本。
