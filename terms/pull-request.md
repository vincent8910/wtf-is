# Pull Request（PR）— 改完請你過目

> **白話說：** PR 就像公司的**公文會簽流程**——你改好了文件，先給主管或同事「看一下」，大家覺得 OK 蓋章了，才正式生效。

---

## 它到底是什麼？

**Pull Request**（簡稱 **PR**）是 [GitHub](github.md) 上最重要的協作功能。

當你在自己的[分支](git.md#分支branch不怕搞砸的實驗場)上改好了文件，你不會直接把修改「塞」進正式版本。而是發一個 PR，意思是：

> 「嘿，我改了一些東西，請你拉過去看看（Pull），看完覺得沒問題就合併吧。」

## 一個 PR 的完整流程

```
1. 你在自己的分支上改好文件
         ↓
2. 到 GitHub 上建立一個 Pull Request
   （寫上你改了什麼、為什麼要改）
         ↓
3. 同事收到通知，打開來看你的修改
   （GitHub 會用紅色和綠色標出刪掉和新增的部分）
         ↓
4. 同事在特定行留言：「這裡要不要改成⋯⋯」
         ↓
5. 你根據意見修改，更新 PR
         ↓
6. 大家都同意了，按下「Merge」合併
         ↓
7. 你的修改正式進入主版本 ✅
```

## 為什麼不直接改？要這麼麻煩？

因為這樣做有三個好處：

| 好處 | 說明 |
|------|------|
| **避免錯誤** | 多一雙眼睛審查，錯字、邏輯錯誤容易被抓到 |
| **記錄討論** | 所有的討論都留在 PR 裡，三個月後還能回頭看當時為什麼這樣改 |
| **安全合併** | 不會不小心覆蓋別人的工作 |

就像重要的合約不會只有一個人看——總是要經過審核流程才簽字。

## PM 會怎麼用到 PR？

### 場景一：修改技術規格

```
你（PM）修改了 API 規格裡的驗證流程
→ 發 PR 給工程師 review
→ 工程師說「這個時間限制要改成 5 秒比較合理」
→ 你修改後，工程師按 Approve
→ 合併，新的規格正式生效
```

### 場景二：審查別人的文件

```
技術寫手提交了一份新的使用者手冊
→ 你收到 PR 通知
→ 你看到第 15 行的描述不太清楚，在那行留言
→ 技術寫手修改
→ 你按 Approve，合併
```

## 怎麼在 GitHub 上建立 PR？

如果你用 [Git 指令](git.md) 把修改 push 上去後：

1. 打開 GitHub 網站，進入你的 Repository
2. GitHub 通常會顯示一個黃色橫幅：「你剛 push 了一個分支，要建立 Pull Request 嗎？」
3. 點 **Compare & pull request**
4. 填寫標題（改了什麼）和描述（為什麼改）
5. 點 **Create pull request**

就這樣！接下來等同事 review。

> 💡 **如果只是小修改**（像改個錯字），你也可以直接在 GitHub 網頁上編輯檔案，它會自動幫你建立 PR。
