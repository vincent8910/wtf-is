# Agile / Scrum（敏捷開發） — 把大專案切成小段，快速交付、快速調整

> **白話說：** 與其花半年閉關做完一個大東西才給客戶看，不如每兩週就交一小段成果，邊做邊調整方向。

---

## 它到底是什麼？

傳統做專案的方式是「瀑布式」：先花很久寫完整的規格 → 然後工程師埋頭做幾個月 → 最後一次交出成品。問題是：等做完才發現方向不對，前面的時間都白費了。

**Agile（敏捷）** 的做法不一樣：把專案切成很多小塊，每隔一段時間（通常兩週）就交一小段成果。如果方向不對，最多浪費兩週，馬上調整就好。

**Scrum** 是 Agile 裡最常見的一套做法，有幾個固定的元素：

### Sprint（衝刺）

一個 **Sprint** 通常是 1～2 週。團隊在每個 Sprint 裡專注完成幾個小任務。Sprint 結束後展示成果、回顧哪裡可以改進、然後開始下一個 Sprint。

### Daily Standup（每日站會）

每天花 10～15 分鐘，每個人回答三個問題：
1. 昨天做了什麼？
2. 今天要做什麼？
3. 有沒有遇到什麼 [Blocker](blocker.md)？

叫「站會」是因為站著開比較快（不會拖太久）。

### Backlog（待辦清單）

所有還沒做的需求都放在 **Backlog** 裡。每個 Sprint 開始前，團隊會從 backlog 裡挑出這次要做的任務。

## 生活比喻 / 實際例子

- **裝修房子**：不是一次畫好全部設計圖再動工，而是先做客廳、住進去用看看、覺得哪裡要改再調整，然後做臥室、再做廚房……
- **邊吃邊點的火鍋**：不是一次全部點完，而是先點幾盤、吃了覺得好再加點、不好就換別的

| 傳統（瀑布式） | Agile（敏捷） |
|---------------|--------------|
| 一次規劃所有細節 | 只規劃近期要做的 |
| 幾個月後才看到成果 | 每 1～2 週就有東西可以看 |
| 改需求代價很高 | 隨時都能調整方向 |
| 像蓋房子，按藍圖施工 | 像做火鍋，邊吃邊調整 |

## 為什麼要知道這個詞？

- 「我們跑 Scrum」→ 團隊用敏捷的方式管理專案
- 「這個排到下個 Sprint」→ 這個功能排到下一個週期才做
- 「Backlog 裡面有 50 個需求」→ 待辦清單上還有 50 個東西等著做
- 「每天早上有 standup」→ 每天有一個簡短的站會同步進度
- 「Sprint review」→ 每個 Sprint 結束時的成果展示會議
