# Code Review（程式碼審查）— 同事幫你檢查作業

> **白話說：** 就像文章寫完請人校稿一樣，程式碼寫完也要請同事看過、確認沒問題才能合併上線。

---

## 它到底是什麼？

Code Review 就是**一個工程師寫完程式碼後，請其他工程師閱讀、檢查、給意見的流程**。

通常的流程是：
1. 工程師寫完功能，發一個 [PR（Pull Request）](pull-request.md)
2. 其他工程師打開 PR，看裡面改了什麼
3. 如果沒問題 → 按「Approve」（核准）
4. 如果有問題 → 留下 comment（意見），請原作者修正
5. 修正完、核准後 → [merge](merge.md) 進主版本

Code Review 的目的不是找碴，而是：
- 抓出 [bug](bug-debug.md) 或潛在問題
- 確保程式碼品質和風格一致
- 讓團隊其他人也了解這段程式碼在做什麼

## 生活比喻 / 實際例子

**文章校稿 / 公文會簽：**
- 你寫完一篇文章 → 請編輯校稿 → 編輯標記「這邊語句不通」→ 你修正 → 完稿
- 工程師寫完程式 → 請同事 review → 同事留言「這邊有 bug」→ 修正 → merge

**你什麼時候會聽到：**
- 「這個 PR 還在等 **code review**。」
- 「你幫我 **review** 一下。」
- 「**Review** 完了，可以 merge。」

## 為什麼要知道這個詞？

- 當工程師說「在等 review」，意思是功能已經寫好了，但要等同事看過才能上線
- 這是「PR → Review → Merge → Deploy」流程中很重要的一環，會影響上線的時間
- Code Review 通常需要幾小時到一兩天，取決於團隊大小和修改內容的複雜度
- 如果你催工程師「功能什麼時候好」，他可能回「寫好了，在等 review」——這時候瓶頸不在他身上
