# Epic（史詩）— 一個很大的功能需求，要拆成好幾個小任務才能完成

> **白話說：** 蓋一棟房子就是一個 Epic——它太大了，不可能一口氣做完，必須拆成挖地基、蓋牆壁、裝水電、做裝潢這些小任務。

---

## 它到底是什麼？

Epic 在專案管理裡是指**一個範圍很大的功能或目標**，大到不可能在一個 [Sprint](sprint.md) 裡做完，必須拆解成好幾個小的 [User Story](user-story.md) 來逐步完成。

舉個例子，假設你們要在 App 裡加「會員系統」，這就是一個 Epic。它可以拆成：

- **User Story 1**：使用者可以註冊帳號
- **User Story 2**：使用者可以登入 / 登出
- **User Story 3**：使用者可以修改個人資料
- **User Story 4**：使用者可以重設密碼
- **User Story 5**：使用者可以綁定社群帳號

每個 User Story 可以在一個 Sprint 裡完成，但整個 Epic（會員系統）可能需要 2～3 個 Sprint。

簡單來說，工作的大小層級是：

```
Epic（史詩）→ 大目標，跨好幾個 Sprint
  └─ User Story（使用者故事）→ 一個具體功能，一個 Sprint 內可完成
       └─ Task（任務）→ 更細的執行項目
```

## 生活比喻 / 實際例子

你決定要「把家裡重新裝潢」——這就是一個 Epic。

你不可能一天搞定，所以拆成幾個階段：
- 第一週：拆掉舊裝潢、清場
- 第二週：水電重拉、泥作
- 第三週：木工、油漆
- 第四週：傢俱進場、收尾

每一週就像一個 Sprint，每週裡面的工作就是 User Story。

**造句**：「『購物車功能』是一個 **Epic**，我們先拆成幾個 User Story，再逐步排進 Sprint。」

## 為什麼要知道這個詞？

- PM 在規劃路線圖（Roadmap）時，通常用 Epic 來代表大功能
- 開會時聽到「這個 Epic 預計什麼時候完成」，就是在問一個大目標的進度
- 知道 Epic → User Story → Task 的層級關係，就不會搞混需求的大小
