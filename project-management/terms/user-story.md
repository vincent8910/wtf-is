# User Story（使用者故事）— 用使用者的角度描述一個功能需求

> **白話說：** 就像點餐時說「我想要一杯不太甜的拿鐵」，而不是跟咖啡師說「用 18g 咖啡粉萃取 30ml espresso 再加 200ml 牛奶⋯⋯」。

---

## 它到底是什麼？

User Story 是一種**寫需求的方式**，重點是「使用者想要什麼」，而不是「技術上要怎麼做」。

它有一個經典的格式：

> **身為**（某種使用者），**我想要**（某個功能），**這樣我就能**（達成某個目的）。

舉個例子：
- **身為**一個線上購物的顧客，**我想要**用 LINE Pay 結帳，**這樣我就能**不用另外輸入信用卡號。

為什麼要這樣寫？因為工程師太容易一頭栽進技術細節，忘了「到底是誰要用」和「為什麼需要這個功能」。User Story 強迫大家先回到使用者的角度思考。

User Story 通常會被放進 [Backlog](backlog.md) 裡，等到 [Sprint](sprint.md) 開始時被挑出來開發。比較大的需求會先寫成 [Epic](epic.md)，再拆成好幾個 User Story。

## 生活比喻 / 實際例子

想像你在裝潢新家，你跟設計師說：

- ❌ 技術語言：「我要在牆壁裝一個 220V 的三孔插座，距地面 120 公分」
- ✅ User Story：「我想在沙發旁邊充手機，不用彎腰」

設計師聽了第二種，就知道重點是「方便充電」，可能提出更好的解決方案，例如裝一個帶 USB 孔的小桌。

**造句**：「這個功能的 **User Story** 是什麼？我想先搞清楚使用者到底要解決什麼問題。」

## 為什麼要知道這個詞？

- PM 寫需求文件時很常用 User Story 格式
- 開 Sprint Planning 會議時，團隊會一起看 User Story 來理解要做什麼
- 它幫助團隊聚焦在「為什麼要做」，而不是一開始就吵「怎麼做」
- 如果你在提需求，試著用 User Story 格式寫，工程師會更容易理解你的意思
