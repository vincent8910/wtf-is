# WebSocket — 讓瀏覽器和伺服器即時雙向溝通

> **白話說：** WebSocket 就像打電話——雙方一直保持連線，有話隨時講。不像傳簡訊（HTTP）每次都要重新撥號。

---

## 它到底是什麼？

一般的網頁請求（HTTP）像傳簡訊：

```
你：「現在幾度？」（發送請求）
伺服器：「28 度」（回應）
＝斷線＝

五分鐘後...
你：「現在幾度？」（重新發送請求）
伺服器：「29 度」（回應）
＝斷線＝
```

每次問一次就掛掉，要知道最新資訊只能一直問一直問。

**WebSocket** 像打電話：

```
你：「嗨，我想一直知道溫度變化」（建立連線）
伺服器：「好，28 度」
伺服器：「變了！29 度」
伺服器：「又變了！27 度」
你：「收到，那我也跟你說，我把冷氣開了」
（雙方一直連著，隨時互相講話）
```

一旦連上，雙方都能隨時傳訊息給對方，不用一直重新連線。

## 生活比喻 / 實際例子

你一定用過 WebSocket，只是不知道：

| 你做的事 | 背後在用 WebSocket |
|---------|------------------|
| LINE 即時聊天 | 對方打字你馬上看到 |
| 看股票即時報價 | 數字一直跳、一直更新 |
| 多人線上遊戲 | 你動一下，別人立刻看到 |
| Google Docs 協作 | 同事打字你即時看到 |

如果這些都用傳簡訊的方式（HTTP），每秒鐘要問幾十次「有新訊息嗎？」，伺服器會被煩死。

**造句：**「聊天室要即時收到訊息，用一般的 [REST](rest.md) 太慢了，得用 **WebSocket** 才行。」

## 為什麼要知道這個詞？

- 任何需要「即時更新」的功能（聊天、通知、股價、遊戲），背後幾乎都靠 WebSocket
- 當 PM 說「這個要即時的」，工程師腦中浮現的第一個技術就是 WebSocket
- 跟 [REST](rest.md) 是互補關係：一般查詢用 REST，即時更新用 WebSocket
- WebSocket 會持續佔用連線資源，所以不是所有功能都適合用——不需要即時的東西，用 REST 就好
