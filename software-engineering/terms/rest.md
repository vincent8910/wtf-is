# REST / RESTful — 軟體之間溝通的設計風格

> **白話說：** REST 就像圖書館的借還書規則——查書用查詢窗口、借書填借書單、還書放還書箱，每件事都有固定的流程和窗口。

---

## 它到底是什麼？

**REST** 是 **Representational State Transfer** 的縮寫，中文翻譯超長不用背，只要知道：

> **REST 就是一套「軟體之間怎麼溝通」的規則。**

你去圖書館的時候，不會跑進書庫自己找書對吧？你會：

```
想查書 → 用電腦查詢系統（GET）
想借書 → 填借書單（POST）
想改預約資料 → 跟櫃台說要修改（PUT）
想取消預約 → 跟櫃台說要取消（DELETE）
```

REST 就是這套規則搬到網路上。每個「資料」都有一個固定網址，你對那個網址做不同的動作，就能查詢、新增、修改、刪除資料。

如果一個系統遵守這套規則，工程師就會說它是 **RESTful** 的——就像一間圖書館有完善的借還書制度，大家來了都知道怎麼操作。

## 生活比喻 / 實際例子

想像你去麥當勞自助點餐機：

| 你想做的事 | REST 的說法 | 麥當勞的動作 |
|-----------|-----------|-------------|
| 看菜單有什麼 | GET /menu | 瀏覽菜單畫面 |
| 點一份大麥克 | POST /orders | 按下「加入餐點」 |
| 改成不要酸黃瓜 | PUT /orders/123 | 修改訂單內容 |
| 取消整筆訂單 | DELETE /orders/123 | 按下「取消訂單」 |

每個動作都有對應的「動詞」和「網址」，清楚明瞭，不會搞混。

**造句：**「我們的 [API](api.md) 是 **RESTful** 的，你用 GET 打這個網址就能拿到會員資料。」

## 為什麼要知道這個詞？

- 幾乎所有 App、網站背後都在用 REST 風格的 [API](api.md) 溝通
- 當工程師說「打一支 REST API」，就是「照規則對某個網址發送請求，拿到資料」
- 前端工程師和後端工程師之間的溝通，大部分靠的就是 RESTful API
- 跟 [WebSocket](websocket.md) 不同——REST 像傳簡訊（一問一答），WebSocket 像打電話（持續連線）
