# GitHub — 程式碼的 Google Drive

> **白話說：** 如果 [Git](git.md) 是你電腦上的「存檔功能」，那 GitHub 就是「雲端硬碟」——讓你的存檔可以上傳、分享、跟別人一起協作。

---

## 它到底是什麼？

**GitHub** 是一個**雲端平台**，讓你把用 Git 管理的檔案上傳到網路上，跟團隊成員共同編輯、追蹤進度、審查品質。

就像 Google Drive，但是專門為「有版本控制需求」的工作設計的。

## Git vs GitHub：一次搞懂

| | Git | GitHub |
|---|---|---|
| **是什麼** | 一個裝在你電腦上的工具 | 一個雲端網站平台 |
| **比喻** | 單機遊戲的存檔功能 | 雲端存檔 + 多人連線 |
| **在哪裡** | 你的電腦本機 | 網路上（github.com） |
| **功能** | 記錄修改、版本控制、後悔藥 | 分享、協作、審查、討論 |
| **離線能用嗎** | ✅ 完全可以 | ❌ 需要網路 |
| **費用** | 免費 | 免費方案 + 付費升級 |
| **一定要一起用嗎** | 可以單獨用 Git 不用 GitHub | GitHub 必須搭配 Git 使用 |

**一句話版：Git 是引擎，GitHub 是公路。引擎讓車能動，公路讓車能去遠方。**

## GitHub 上的重要概念

### Repository（倉庫）— 專案的家

一個 Repository（簡稱 **Repo**）就是一個專案的所有檔案集合。就像 Google Drive 上的一個共用資料夾。

```
我的公司/
├── product-spec（產品規格的 repo）
├── marketing-site（行銷網站的 repo）
└── internal-tools（內部工具的 repo）
```

### Pull Request（PR）— 改完請你過目

PR 是 GitHub 最重要的協作功能。流程是這樣的：

```
1. 你在自己的分支上改好文件
2. 提交一個 Pull Request（「請你看看我改了什麼」）
3. 同事看了你的修改，留言討論或建議
4. 大家覺得 OK，合併（merge）進主分支
```

就像你寫了一份公文，先給主管「會簽」，主管看完蓋章才正式發出。

> 💡 **PR 的好處：** 不是直接改正式版本，而是先讓別人 review。這樣可以避免錯誤被直接推上線。

### Issue（問題追蹤）— 待辦事項

GitHub Issues 就像專案的**待辦清單**。你可以用它來：

- 回報 bug：「登入頁面按鈕壞了」
- 提出需求：「希望新增匯出 CSV 功能」
- 追蹤進度：指派給特定的人、設定截止日

就像 Trello 或 Jira 的簡易版，但直接跟程式碼放在一起。

### README — 專案的門面

每個 Repo 都會有一個 `README.md` 檔案，就是你進入 GitHub 專案頁面時看到的第一份文件（你現在讀的這個專案就有一個）。

它告訴訪客：這個專案是什麼、怎麼安裝、怎麼使用。

## PM 為什麼需要學 GitHub？

### 場景一：技術規格跟程式碼一起管理

```
以前的流程：
PM 在 Google Docs 寫規格 → 丟連結給工程師 → 工程師做完了
→ PM 更新 Docs → 但工程師看的還是舊版 → 混亂 💥

用 GitHub 的流程：
PM 在同一個 repo 寫規格 → 工程師看到直接開工
→ PM 更新規格用 PR → 工程師收到通知 → 永遠同步 ✅
```

### 場景二：用 Issue 追蹤需求

```
PM 建立 Issue：「#42 使用者希望可以匯出 PDF」
→ 指派給工程師小明
→ 小明開分支處理，PR 裡寫「fixes #42」
→ PR 合併時 Issue 自動關閉
→ PM 不用追著人問「做好了沒」
```

### 場景三：用 PR 審查文件品質

```
技術寫手提交 PR：「更新 API 文件 v2」
→ PM 看到哪些行被改了（紅色刪除、綠色新增）
→ PM 在特定行留言：「這段描述不太清楚，能改成…嗎？」
→ 技術寫手修改後，PM 按 Approve
→ 合併！
```

## GitHub 常用操作

### 在網頁上就能做的事（不用開 Terminal）

| 操作 | 怎麼做 |
|------|--------|
| 看檔案 | 直接在 GitHub 頁面點開檔案 |
| 小幅修改 | 點檔案右上角的鉛筆圖示 ✏️ 直接編輯 |
| 建立 Issue | 點 Issues 分頁 → New issue |
| 看 PR | 點 Pull requests 分頁 → 看修改內容 |
| 留言討論 | 在 PR 或 Issue 下方打字就好 |

> 💡 **如果你只是偶爾改文件，用 GitHub 網頁就夠了，不一定要用 Terminal。** 但如果要頻繁改大量文件，學 [Git 指令](git.md) 會快很多。

### 需要在 Terminal 做的事

| 操作 | 指令 | 說明 |
|------|------|------|
| 把 repo 下載到電腦 | `git clone 網址` | 第一次拿到專案時用 |
| 上傳修改 | `git push` | 把本機的存檔推上 GitHub |
| 下載別人的最新修改 | `git pull --rebase` | 更新到最新版本 |
